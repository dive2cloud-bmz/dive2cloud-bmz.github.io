MVCC

老生常谈了，简单说一下

事务隔离级别

  1. RU：脏读，一个事务可以读到另一个事务未提交的数据
  2. RC：读已提交，一个事务可以督导其他数据已提交的事务
  3. RR：多个事务执行过程中是独立的，不会被其他事务影响，等到互相提交的时候才可能冲突
  4. SERIAL：序列化，串行执行，全部一致

常见并发控制

  1. MVCC：多版本并发控制
  2. S2PL：严格2阶段锁，读写互斥保证串行
  3. OCC：乐观并发控制，冲突就回退当前事务

MVCC 基本是读写不互斥的基准了

MYSQL MVCC

innodb 中是通过 隐藏字段 + undo log chain + read view 三个概念来实现。

隐藏字段

  1. TRX_ID：最近一次修改这行的事务ID
  2. ROLL_PTR：回滚指针，指向一个 undo log
  3. ROW_ID：可选，生成聚簇索引键

undo log

  1. 每次 update 或者 delete 不是直接覆盖原来的数据，而是将旧版本写一个 undo log，更新这行的 trx_id 和 db_roll_ptr
  2. undo log 保存了这行的历史版本链

    当前行 → DB_ROLL_PTR → Undo Log 版本1 → DB_ROLL_PTR → 版本2 → …

read view

当一个事务执行 SELECT 时，会创建一个 Read View，记录系统当前的事务状态
这个 Read View 里记录了当前事务能看到哪些行的那些版本
根据设置的数据隔离级别，Read View 创建的时机是不一样的
比如设置成RC，那就是每次SELECT 开始创建 view
设置成 RR，那就是事务第一次 SELECT 时创建，后面一直复用

清理

  1. undo log 不能随便清除
  2. 当所有 read view 都不需要某个版本 undo log 时，有一个后台进程清理 undo log
  3. 如果有长期运行的事务，会导致 undo log 一直不清理，可能打满磁盘容量

PG MVCC

PG不像 MYSQL 有 undo log，而是天生数据就是多版本的。
这个设计让每次更新或删除不是去修改数据，而是增加一条新数据，然后对老数据标记 “过期”

有点 HBASE FSM TREE 那味儿了

page 页


一个 block 里有多个 page，page 默认大小是 8K，当一个 page 写满之后就会递增一个 page 号，page 都是从0号开始

pg 单表至多 32T 数据，因为 pg 默认使用 32 位寻址，最多有 2^32 个 page

header： 24B，记录元信息
line pointers: 行指针，每个 4B
heap tuples：真正的行数据


多版本是靠这样的代码来维护的，介绍一下这里的字段

  1. t_xmin：插入当前元祖的事务 txid
  2. t_xmax：删除或更新tuple的 txid，没删除的时候是 0
  3. t_cid：包含了 cmin 和 cmax 两个字段，分别是创建 tuple 的 cid 和删除 tuple 的 cid
  4. t_ctid：第一次创建 tuple 时是自己，假如删除了，会写到新的 tuple

存储上的原理

写新数据时，不删除旧数据，只是标记出来，然后把新数据插入进来
比如我有一个表 test，里面只有一个 id int 字段
insert into test values(1);
会生成一个 tuple，可以用隐藏字段查表
page_test=# select ctid, xmin, xmax, cmin, cmax,id from test;
ctid | xmin | xmax | cmin | cmax | id
——-+——–+——+——+——+—-
(0,1) | 168693 | 0 | 0 | 0 | 1

当我执行 update test set id=2;之后再去查表
page_test=# select ctid, xmin, xmax, cmin, cmax,id from test;
ctid | xmin | xmax | cmin | cmax | id
——-+——–+——+——+——+—-
(0,2) | 168702 | 0 | 0 | 0 | 2

查出来 ctid 和 xmin 都变化了
我在当前库安装 pageinspect 插件，可以进一步的看信息
CREATE EXTENSION IF NOT EXISTS pageinspect;
查询具体 page 信息：
select * from heap_page_items(get_raw_page('test',0));
看到

那么 lp = 1 的记录就是第一次 insert 的结果，t_xmax = lp=2 的 t_xmin，代表这个 tuple 当前被这次事务删除了
所以此时 tuple lp1 就变成了 dead tuple，此时执行 VACUUM 会清理掉这个 tuple

但是为啥有一个啥也没有的 page，这要研究一下 vacuum 了，嘻嘻

总结一下存储上怎么做的：
pg 用 header 和隐藏字段给每个 tuple 记录了版本号，版本从旧到新会形成一条版本链(ctid 在删除时会指向新 tuple）
用 vacuum 可以清理掉没用的 tuple

好处

  1. 无论事务进行了多少操作，事务都可以立刻回滚
  2. 数据可以连续更新，不需要关注 innodb 保证回滚段没用完等限制

坏处

  1. 旧版本数据得用额外机制清理
  2. 旧版本数据可能会导致查询扫描的数据块变多，查询慢

事务快照

对标 innodb 的 read view，根据隔离级别确定 tuple 可见性
快照也有 xmin 和 xmax，但是含义和上面 tuple 的不一样
创建一个快照的过程

  1. 查出所有没提交并活跃的事务
  2. 找到1里最小的 xid，记录到快照的 xmin
  3. 找到1里最大的 xid，加1后存到 xmax
  4. 根据隔离级别设置不同的 satisfies，创建事务快照
  • 当事务 ID 小于 xmin 就可以认为是已经提交的，这些 tuple 都可见
  • 当事务 ID 大于 xmax 就代表是另外的正在执行的事务B，B的修改当前快照看不见
  • 当事物的ID 在 [xmin, xmax) ，需要结合 clog 判断（也就是 satisfies）

后面再学习一下 clog 吧，嘻嘻

隔离级别怎么决定快照创建的

  1. 对于 RC 和 RUC：每个 query 都生成一个快照（还有说 PG 里就没有读未提交的）
  2. RR：所有 query 都用第一次 query 创建的相同的快照
  3. SERIAL：锁，和这些无关了
